Introduction

JMFFT is a library of Fourier transform subroutines written by Jean-Marie Teuler, emulating most of the Fourier transform subroutines from the Cray library (the SCILIB).

JMFFT offers the following advantages:

  • JMFFT is very complete: it covers most of the SCILIB CRAY subroutines, including some obsolete versions.
  • JMFFT is very portable.
  • JMFFT is very easy to use because it exactly emulates the call sequences of the SCILIB CRAY: same arguments and same dimensions.
  • Although this was not the major concern during its design, JMFFT performs reasonably well.
  • JMFFT can be optimized quite easily for a given architecture because all the computational time is spent in a small number of loops of around 30 rows.

Restrictions

  • JMFFT can a priori process arrays sized with any combination of prime numbers. That said, the version provided is limited to the combinations of the following prime numbers: 2, 3 and 5 on the one hand (for which particular subroutines are written for performance reasons) then a list of other prime numbers. This list is made up of the following prime numbers: 7, 11, 13, 17, 19, 23 and 29. It is quite easy to extend this list. You just have to modify the following instructions in the jmfact subroutine :
      integer, parameter :: nfirst = 7
  integer, dimension (0: nfirst-1) :: prime = (/ 7,11,13,17,19,23,29 /)
    so as to include the prime numbers of its choice, and to recompile this subroutine.

  • Another limitation, for which there is unfortunately no solution for the moment, concerns the real-complex and complex-real transforms. These are available, but require at least one of the dimensions to be even:
    • in 1D, the dimension of the TF must be even
    • in 2D, one of the two dimensions must be even
    • in 3D, one of the three dimensions must be even
    • in multiple 1D, the dimension of the TF or the number of TF must be even.

  • Despite the choice of algorithms very oriented towards vectorization, the performances do not yet equal those of the large libraries, in particular the manufacturer libraries. That said, if someone wants to get started, their task will be greatly facilitated by the fact that the computing time is concentrated in a few subroutines: jmccm1d2 (base 2, unused in principle), jmccm1d3 (base 3), jmccm1d4 (base 4), jmccm1d5 (base 5) and jmccm1dp (general prime number).

  • Although the Fortran version is written in Fortran 90, JMFFT does not take advantage of the many possibilities of this language, which would nevertheless be very interesting here (dynamic allocation of memory, modules, tables with implicit profile, optional arguments, etc.). The essential reason is the compatibility with the SCILIB CRAY: the user must be able to replace his calls to the SCILIB directly by calls to JMFFT without changing anything else, which excludes any sophisticated use of the Fortran 90 possibilities.

  • JMFFT performs a number of checks on the arguments passed to it. It can thus detect errors. The problem is then to inform the user. It is not that simple because the SciLib did not include a return code among the arguments of its subroutines and that JMFFT, for compatibility reasons, could not add one. That said, a solution has been found to circumvent this gap. See below .

Control of JMFFT

There are service routines in JMFFT that control its behavior.
  1. JMSETNWORK
    This subroutine allows you to control more finely the working space available to the 2D and 3D subroutines. In fact, in the SCILIB, the WORK table must be sized at 512 * MAX (N1, N2) in 2D and at 512 * MAX (N1, N2, N3) in 3D. With such a dimension, the length of the loops in the calculation subroutines is 128, which is indeed optimal on the C90s. On the other hand, such a loop length is far from being optimal on a Fujitsu VPP300 or on a NEC SX-5 for example. JMSETNWORK allows to provide more workspace to JMFFT, and thus to obtain better performances. Its use is very simple. For example in the following call: 
      call jmsetnwork (4096)
    4096 workspace items are provided to JMFFT.
    Notes:
    • Caution: The user must take care of the coherence between the dimension supplied to JMFFT and the dimensioning of the WORK table in his calling subroutine. JMFFT does not have any means to know the real dimension of this array (this is also why if the user oversizes it, he must inform JMFFT of it by calling JMSETNWORK ). In particular, the actual dimension of the WORK array must always be greater than or equal to what JMFFT expects (whether it is the default value or the value passed to it through JMSETNWORK ).
    • This mechanism was designed to increase the available workspace, but it can also be used to reduce it (in case of running out of memory resources).
    • This mechanism is only effective on the 2D and 3D subroutines.
    • This workspace will be used in all the 2D and 3D subroutines that follow, and will remain in effect unless we call JMSETNWORK again .

  2. JMSETERREUR
    The JMFFT subroutines check the consistency of the arguments transmitted. When they detect an inconsistency, the default behavior is to stop everything by displaying an error message and triggering a STOP statement . This behavior is imposed by the compatibility with the SCILIB which has not provided for any return code type argument , like the one found for example in NAG and IMSL. That said, we can modify it slightly. If we make the following call: 
      call jmseterror (.false.)
    in the event of an error, the execution is not interrupted. See below the two subroutines that allow you to check if an error has occurred and if so which one.

  3. JMGETCODE
    If the previous call has been made, in the event of an error the execution continues. It is therefore up to the user to check, after each call to JMFFT, whether an error has occurred. This can be done in the following way: 
      call jmgetcode (irc)
    where irc is an integer variable that contains the return code from the last call to JMFFT. If this code is null, then everything went well. Otherwise, an error has been detected. To find out which one, see below.

  4. JMGETMESSAGE
    If an error occurred and that one obtained the error code irc by call of JMGETCODE , one can recover the corresponding error message by the following call: 
      call jmgetmessage (irc, message)
    where message is a character string variable which will contain the clear text of the error message on return.
